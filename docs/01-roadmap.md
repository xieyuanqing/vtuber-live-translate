# 开发路线图（详细版）

每一步都按同一格式写：**目标 → Claude 做什么 → 你做什么 → 验收标准 → 常见坑**。验收通过才进下一步，结果记到 [04-dev-log.md](04-dev-log.md)。

看不懂的名词先查 [03-android-primer.md](03-android-primer.md)，里面有本项目用到的所有安卓概念和名词表。

## 开发工作流（每一步都这么转）

```text
Claude 在电脑上写代码 → 命令行编译出 APK
→ 你把 APK 装到手机（两种方式，见下）
→ 你按"你做什么"清单操作、对照验收标准
→ 把现象告诉 Claude（正常/异常、截图、报错）
→ 通过 → 记 dev-log，进下一步；不通过 → 修
```

装 APK 的两种方式：

| 方式 | 操作 | 说明 |
|---|---|---|
| **USB + adb（推荐）** | 手机连电脑、开 USB 调试，Claude 直接命令行安装 | 出问题时 Claude 能自己看手机日志（logcat），调试效率高很多 |
| 手动拷贝 | 把 APK 文件传到手机（微信/网盘/数据线均可），点开安装 | 不用插线，但出问题要靠你描述现象 |

> 建议开发时段尽量插着 USB。第一次连接手机会弹"允许 USB 调试吗"，勾选"一律允许"。

## 技术基线（定死，不再纠结）

| 项 | 选择 | 理由 |
|---|---|---|
| 语言 | Kotlin | 官方主推 |
| UI | **全部用传统 View**（放弃之前写的 Compose） | 本 App 界面很简单；悬浮窗本来就必须用 View；少一层版本兼容矩阵，编译报错对新手更好懂 |
| minSdk | 29（Android 10） | AudioPlaybackCapture 内录的最低系统要求 |

| 构建 | Gradle 8.9 + AGP 8.7.3 + Kotlin 2.0.21 | 已验证的稳定版本组合 |
| 网络 | OkHttp WebSocket | 成熟；Base URL 做成可配置（代理/反代用） |
| 包名 | `com.xyq.livetranslate` | — |
| 项目位置 | `D:\vtuber-live-translate\android\` | 用 Android Studio 打开的就是这个目录 |

## Step 0.5：构建链路验证（Claude 侧）

**目标**：证明 Claude 能在这台电脑上用命令行把安卓项目编译成 APK——这是后面所有步骤的工作方式。

- **Claude 做**：搭最小项目骨架（一个只显示一行字的 App），命令行编译出 `app-debug.apk`
- **你做**：把 APK 装到手机，打开看到文字
- **验收**：手机上能打开这个骨架 App
- **常见坑**：首次编译要从网上下载依赖（几百 MB，走 dl.google.com，国内一般可直连），可能要等几分钟

## 阶段 0：四个验证件（把核心风险从大到小逐个拆掉）

> **2026-07-08 决策变更**：用户确认跳过 Step 1–3 的独立 demo，直接实现 Step 4 整合版（v0.2）。依据：手机上腾讯会议等第三方 App 内录已验证可行。若整合版某环节出问题，再回退到分步排查。

### Step 1：内录 demo —— 能抓到 YouTube 的声音吗

**目标**：真机确认第三方 App 能通过 AudioPlaybackCapture 抓到 YouTube 音频数据。

**Claude 做**：

- `MainActivity`：一个"开始/停止捕获"按钮 + 实时音量电平条 + "录 10 秒样本"按钮
- `CaptureService`：前台服务（挂通知栏那种），持有 MediaProjection 授权，用 AudioRecord 循环读系统播放音频
- 10 秒样本存成 **WAV 文件**（带文件头的 PCM，手机、电脑都能直接播放）
- manifest 里声明：`RECORD_AUDIO`、`FOREGROUND_SERVICE`、`FOREGROUND_SERVICE_MEDIA_PROJECTION`、`POST_NOTIFICATIONS`

**你做**：

1. 装 APK，打开，点"开始捕获"
2. 依次同意弹出的权限：通知权限 → 录音权限 → 系统的"开始录制或投放屏幕"确认框（这就是 MediaProjection 授权，**每次开始捕获都会弹**，正常现象）
3. 切到 YouTube 放一段日语视频
4. 切回本 App（或分屏），看电平条是否随声音跳动
5. 点"录 10 秒样本"，回放听一下（或拷到电脑听）

**验收**：电平条随 YouTube 声音明显跳动；10 秒 WAV 回放能清楚听到 YouTube 内容。

**常见坑**：

- 电平恒为 0：YouTube 没在播 / 手机静音了（内录抓的是解码后的音频流，一般不受媒体音量影响，但要排除）/ ROM 限制——真出现再排查
- 通知栏没有常驻通知：前台服务没起来，ROM 杀了——去电池优化白名单里放行本 App
- MediaProjection 弹窗点了"取消"：重新点开始就行

### Step 2：悬浮窗 demo —— 字幕能盖在 YouTube 上吗

**目标**：悬浮窗能持续显示在 YouTube 全屏播放之上，可拖动，不被杀。

**Claude 做**：

- 悬浮窗权限引导（跳到系统设置的"显示在其他应用上层"页面）
- `SubtitleOverlay`：WindowManager 悬浮窗，半透明黑底白字，显示测试文字（每秒变化的计数器，方便确认它活着），手指可拖动

**你做**：

1. 按 App 引导开启悬浮窗权限（每个 ROM 的设置页面长得不一样，找"悬浮窗"或"显示在其他应用上层"）
2. 打开悬浮窗，切到 YouTube 全屏放视频
3. 拖动悬浮窗换位置，等 10 分钟看它是否还在、计数器是否还在走

**验收**：YouTube 全屏时悬浮窗可见、可拖动；挂 10 分钟不消失、计数不停。

**常见坑**：

- 全屏视频时悬浮窗不见了：部分 ROM 对全屏场景有额外的悬浮窗限制，去 App 权限设置里找"后台弹出界面"/"全屏显示悬浮窗"类开关
- 锁屏再解锁后悬浮窗没了：service 被杀，电池白名单

### Step 3：Gemini Live WebSocket demo —— 手机能连上并收到中文吗

**目标**：从手机直连 Gemini Live Translate，推测试音频，收到中文转写。**这一步同时暴露代理问题。**

**Claude 做**：

- 简单设置界面：填 API key（暂存本地，加密放阶段 1）
- `GeminiLiveClient`：OkHttp WebSocket，setup 消息严格按 [02-tech-notes.md](02-tech-notes.md) 的结构
- 把 **Step 1 录的那段 YouTube 日语 WAV** 内置为测试音频（一箭双雕：既是真实样本，又验证格式转换），重采样到 16kHz 单声道推上去
- 屏幕上滚动显示收到的日文转写 + 中文翻译

**你做**：

1. 确认手机代理对本 App 生效（分应用代理勾上）
2. 填入**新申请的** API key，点"测试连接"
3. 看屏幕输出

**验收**：手机屏幕上出现测试音频对应的中文文本。

**常见坑**：

- 连接超时：九成是代理没对本 App 生效。解决不了就上 VPS wss 反代方案（tech-notes 网络一节）
- close 1007：setup 消息字段位置错（照 tech-notes 抄就不会）
- 401/403：key 错了或没生效

### Step 4：最小闭环 —— 串起来，能看直播了

**目标**：内录 → 翻译 → 悬浮字幕全链路跑通，能撑过一场直播的时长。

**Claude 做**：

- 把 1/2/3 串成完整管线：`CaptureService` 内 AudioRecord → 重采样 16k → 100ms 分块 → WebSocket → 悬浮窗显示（原始输出，先不做稳定器）
- 8 分 30 秒主动重连（新旧连接交接，音频重叠 1–2 秒）+ 异常断线立即重连
- 硬编码一个你常看主播的 prompt（告诉我主播名字，我照原计划 6.3 节的样式写）
- 悬浮窗加个小状态点：绿=已连接，黄=重连中，红=断开

**你做**：

1. 开一场真实直播（或找个存档当直播放）
2. 开始翻译，切 YouTube，连续看 20 分钟以上
3. 留意：字幕是否跟得上、8 分半左右重连时字幕断多久、发热耗电情况

**验收**：连续 20 分钟字幕不断（重连造成的短暂停顿 ≤ 几秒可接受）；直播结束手动停止，App 不崩。

**常见坑**：

- 字幕碎、有重复：**预期内**，阶段 1 的稳定器解决，这步不管
- 静音段字幕长时间不动：正常，主播没说话
- 定时重连后字幕丢一句：重叠缓冲没调好，反馈给 Claude 调

> **到这里 App 已经"能用"。以下是把"能用"变"好用"，按对观感的影响排序。**

## 阶段 1：自用 MVP（顺序可调，每条独立验收）

1. **字幕稳定器**：碎片合并、连续重复去重、临时行/确认行两级显示。验收：同一段直播开/关稳定器对比，明显更连贯 ——**v0.3 已实现，待实测**
2. **悬浮窗控制**：字号、背景透明度、显示行数（1–3）、暂停/继续、收起成小圆点。验收：各控件生效，收起后点圆点能展开
3. **设置页**：API key 加密存储、自定义 API Base URL、目标语言。验收：改 Base URL 指向无效地址会报连接错误（说明生效），改回来恢复
4. **prompt 预设**：本地存多个主播的 prompt 文件，开播前下拉选一个。验收：切换预设后新会话用的是新 prompt
5. **transcript 保存**：日文转写 + 中文输出按日期存 Markdown，可分享出去。验收：看完一场，文件内容完整、时间戳合理
6. **多 key 轮换**（可选）：逗号分隔多个 key，会话随机选。验收：日志能看到不同会话用了不同 key

> **2026-07-08 深夜**：第 1–6 项已全部在 v0.3 / v1.0 实现（暂停/继续用"点击收起小圆点"替代；目标语言固定 zh 未做选择器），待用户实测验收。细节见 dev-log。

## 主要风险与应对

| 风险 | 应对 |
|---|---|
| 单连接约 590 秒被 GoAway 断开 | Step 4 的 8:30 主动重连（已实测确认存在） |
| 字幕碎片化 | 阶段 1 稳定器；阶段 0 期间先忍受 |
| ROM 杀后台 / 悬浮窗 | 前台服务 + 电池白名单；Step 1/2 就会暴露 |
| 直播静音段长时间无返回 | 正常行为，不当断线处理 |
| 免费配额撞限 | 多 key 轮换；留意用量 |
| 手机连不上 Google | Step 3 暴露；分应用代理 → 不行就 VPS wss 透传 |
